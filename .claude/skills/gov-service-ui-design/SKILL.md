# Gov Service UI Design

## 목표

공공 API와 MCP 응답을 시민이 익숙하게 느끼는 정부서비스 화면으로 렌더링한다. 핵심은 정부24를 복제하는 것이 아니라, 통합검색과 서비스 카드 중심의 신뢰 가능한 공공서비스 UX를 만든다.

## 설계 기준

1. 첫 화면은 통합검색을 중심으로 구성한다.
2. 자주 찾는 서비스는 시나리오 탭보다 우선한다.
3. 결과는 `요약 -> 신청/조회/안내 후보 -> 지표/표 -> 출처/상태 -> 안전고지` 순서로 배치한다.
4. 민원서비스, 혜택알리미, 생활, 정책정보 분류를 카드에 명시한다.
5. 각 결과는 출처, 갱신일, 응답 상태를 추적할 수 있어야 한다.
6. 실패한 API는 숨기지 않고 `캐시`, `대체`, `오류` 상태로 표시한다.
7. 로그인, 본인인증, 자동 제출은 구현하지 않는다. 필요한 경우 공식 경로 안내까지만 한다.
8. 모바일에서는 좌측 rail을 상단 안내 또는 접이식 요약으로 재배치한다.
9. 키보드 포커스, heading 계층, 44px 이상 터치 타깃, 상태 변화 안내를 유지한다.
10. 사용자가 정부24 참고 이미지를 제공하면 색, 여백, 카드 radius, 헤더 높이, 메뉴바, 검색창 비율, 오른쪽 로그인 카드, 메가메뉴를 우선 맞춘다. 새 장식보다 실제 스크린샷의 밀도와 공간감을 따른다.
11. 아이콘은 외부 이미지 로딩에 의존하지 않는 inline SVG 컴포넌트를 기본으로 사용한다. GenUI JSON의 `icon` 값이 알 수 없는 문자열이어도 라벨 기반 fallback으로 정부서비스에 어울리는 아이콘을 표시한다.

## 구현 체크리스트

- 통합검색 입력 영역이 첫 viewport에 보이는가?
- 자주 찾는 서비스 또는 추천 질의가 검색 근처에 있는가?
- 서비스 카드는 행동 CTA와 상태 배지를 포함하는가?
- 출처/상태가 화면 하단에만 고립되지 않고 결과 신뢰성을 보강하는가?
- API 실패/부분 실패를 사용자가 이해할 수 있는 문장으로 표시하는가?
- 모바일 1열에서 검색, 서비스 후보, 출처 순서가 자연스러운가?
- JSON으로 들어온 서비스/카테고리 icon 값이 placeholder 없이 실제 아이콘으로 렌더링되는가?

## 테스트 시나리오

- 생활안전 브리핑: 정상 + 캐시 API가 섞인 경우
- 복지서비스 탐색: 일부 API fallback이 발생한 경우
- 무역 데이터 분석: 표와 지표가 중심인 경우
